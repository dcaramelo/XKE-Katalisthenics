Katalisthenics

Object Catalisthenics, les régles :

  1. Un seul niveau d'indentation par méthode
  2. Ne pas utiliser le mot clé ELSE
  3. Encapsuler tous les types primitifs et String
  4. Utiliser les collections first class
  5. Un seul point par ligne
  6. Pas d'abréviation
  7. Limiter la taille des entités
  8. Deux attributs de class maximum
  9. Pas de Getter/Setter ni Properties

Régle 1 : Un seul niveau d'indentation par méthode

a method with a lot of different indentation is harder to understand than a method with less indentation each level of indentation typically means you're doing another thing. => Single Responsibility Principle 
class Board {
    public String board() {
        StringBuilder buf = new StringBuilder();

        // 0
        for (int i = 0; i < 10; i++) {
            // 1
            for (int j = 0; j < 10; j++) {
                // 2
                buf.append(data[i][j]);
            }
            buf.append("\n");
        }

        return buf.toString();
    }
}
 
class Board {
    public String board() {
        StringBuilder buf = new StringBuilder();

        collectRows(buf);

        return buf.toString();
    }

    private void collectRows(StringBuilder buf) {
        for (int i = 0; i < 10; i++) {
            collectRow(buf, i);
        }
    }

    private void collectRow(StringBuilder buf, int row) {
        for (int i = 0; i < 10; i++) {
            buf.append(data[row][i]);
        }

        buf.append("\n");
    }
}

Régle 2 : Ne pas utiliser le mot clé ELSE

Some people think that the if branch should be the every thing is good branch and the else branch being the error case. Others think the if branch should be as short as possible since it makes it easier to understand what would bring us into the else branch. But this rule is about putting you into a situation where you have to consider other ways to implement your conditional logic. Early return, Variable, Polymorphism and the Null Object pattern being two good alternatives.

Régle 3 : Encapsuler tous les types primitifs et String

Primitive Obsession anti-pattern

C'est l'utilisation des primitifs pour représenter des objet métier, comme le fait d'utiliser un String pour représenter un Message ou un Integer pour représenter un Montant

Solution: ValueObject

Si vous devez faire des opérations sur la variable primitif, vous devez l'encapsuler

Régle 4 : Utiliser les collections first class

Ne pas utiliser par exemple des List<String> users mais plutôt une classe plus explicite, comme par exemple Users users

Chaque class contenant une collection ne doit pas contenir un autre attribut

Régle 5 : Un seul point par ligne

Basically, the rule says that you should not chain method calls. However, it doesn't apply to Fluent Interfaces and more generally to anything implementing the Method Chaining Pattern (e.g. a Query Builder).

For other classes, you should respect this rule. It is the direct use of the Law of Demeter, saying only talk to your immediate friends, and don't talk to strangers.

buf.append(loc.current.representation.substring(0, 1));

Régle 6 : Pas d'abréviation

You may answer that it is because you write the same name over and over again? And I would answer that this method is reused multiple times, and that it looks like code duplication.

So you will say that the method name is too long anyway. And I would tell you that maybe your class has multiple responsibilities, which is bad as it violates the Single Responsibility Principle.

I often say that if you can't find a decent name for a class or a method, something is probably wrong. It is a rule I use to follow while designing a software by naming things.

Don't abbreviate, period.

Régle 7 : Limiter la taille des entités

No class over 50 lines and no package over 10 files.

The idea behind this rule is that long files are harder to read, harder to understand, and harder to maintain.

Régle 8 : Deux attributs de class maximum

The main question was Why two attributes? My answer was Why not? Not the best explanation but, in my opinion, the main idea is to distinguish two kinds of classes, those that maintain the state of a single instance variable, and those that coordinate two separate variables. Two is an arbitrary choice that forces you to decouple your classes a lot.

Régle 9 : Pas de Getter/Setter ni Properties

Tell, don't ask.

It is okay to use accessors to get the state of an object, as long as you don't use the result to make decisions outside the object. Any decisions based entirely upon the state of one object should be made inside the object itself.
